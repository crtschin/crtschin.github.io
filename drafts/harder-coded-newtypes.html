There‚Äôs quite a few fun and nifty things you can do in Haskell to avoid developer error. One notable one is the usage of newtype, which some other languages have adopted. It avoids parameter blindness and ensures values are only used in specific locations.

The trick I‚Äôd like to show right now, is to use DataKinds (or TypeData) to achieve something similar using type-level tag values. An idea fleshed out quite well in this paper, which has a sick name (Ghosts of Departed Proofs), by Matt Noonan.

The Example

The easiest example I can think of, is none other than this blog! In the Hakyll setup of this blog, I add metadata to some posts. Two markers I use for example, are published, to indicate whether the post should be visible, and includetoc, to include a generated table of contents, partly using the general flow described in this blogpost by Riccardo Odone.

# markdown post. Hakyll parses this section as yaml.

Both of the metadata fields are booleans

. In my case, I want to arbitrarily accept true-ish values, case-insensitively matching yes or true. But simply returning booleans ‚Äòas is‚Äô, loses the metadata key associated with the value.

checkPublished' :: String -> Metadata -> Bool

  case lookupString "published" meta of
    Nothing -> False
    Just t -> map toLower t `elem` ["yes", "true"]

So‚Ä¶ Newtype?

The simplest alternative is to create a newtype for specifically the publish boolean and to explicitly keep that separate. 
newtype IsPublished = IsPublished {isPublished :: Bool}

checkPublished'' :: String -> Metadata -> IsPublished
checkPublished'' key meta =
  case fmap Text.pack (lookupString "published" meta) of
    Nothing -> IsPublished False
    Just t -> IsPublished $ Text.toLower t `elem` ["yes", "true"]

filterPublished' :: [(IsPublished, Post)] -> [Post]
filterPublished' = mapMaybe $
  \(flag, post) ->
    if isPublished flag
      then Just post
      else Nothing

But this now becomes problematic if I now need similar logic for my includetoc metadata flag, do I copypaste everything? Do I parameterize everything with a (Bool -> a) parameter that encompases the newtype I‚Äôll add for each flag? That sounds esoteric and horrible!

No, Phantoms!

A solution I like here, is akin to using TypeData with phantom tags that serve as proof witnesses of the checking function. That‚Äôs a lot of abstract type jargon, that serves no one, so let‚Äôs break it down.

TypeData is the extension that allows you to define type level values. The textbook case is type level naturals via type data Nat = Zero | Succ Nat. For our purposes, we‚Äôll use it to create small tags we can pass around. 
type data Published = IsPublished
type data TableOfContents = IncludeTableOfContents

Phantom types are the fun idea of only carrying around knowledge via variables on the type level, with no equavalent on the value level. It‚Äôs a technique that‚Äôs used quite commonly in Haskell. In my case, I want to attach my tags to booleans, but only on the type level. 
newtype Flag a = Flag {getFlag :: Bool}

I didn‚Äôt change the underlying type, it‚Äôs still a Bool. I only added a type variable. One nice observation is that my Flag IsPublished is isomorphic to to the my IsPublished newtype I defined a bit ago.

Now I can use this flag to define generic functions over my flag types, and define ‚Äúsmart‚Äù constructors that introduce these type level constructors as witnesses of the fact that the flag was created correctly
Without avoiding exporting the Flag constructor, this unfortunately doesn‚Äôt proclude someone manually creating a proof witness incorrectly. The Flag constructor essentially plays the role of QED in our functions, so incorrect usage means you introduce incorrect values, subverting why we‚Äôre doing this in the first place.

A Bit Contrived

So I‚Äôve discussed how to easily handle boolean flags in post metadata, but the same approach can be extended to other types of data as well. Say I wanted to support guest authors and put their names under titles in posts, but only when a guest author is actually relevant (otherwise it‚Äôs me!). 
---
ispublished: false
includetoc: true
author: Billy Bob Barker (fictional)
---

If I follow similar line of thinking as the previous section. I‚Äôd build a wrapper akin to newtype Attribute a = Attribute (Maybe String), but then have almost, but not quite the same logic as I do for Flag. This code duplication is again, kind of an eyesore. So let‚Äôs create something a bit more generic.

In identifiable pattern here is that datatype encodes a value whose construction is checked against something (that something could also be a hardcoded value written by hand, thereby going through some mental hueristic). I can write that as a datatype, whose specialization are both the Attribute and Flag types. 
-- Generalizes both `Flag` and `Attribute`, and any other value type with an
-- additional phantom type variable.
newtype CheckedValue a b = CheckedValue {getCheckedValue :: a}

-- These are isomorphic to my original phantom newtype wrappers.
type Flag a = CheckedValue Bool a
type Attribute a = CheckedValue (Maybe String) a
-- A tag I'll use for the guest-authorship.
type data Author = GuestAuthor

-- I can write smart helper that specialize the type variables in CheckedValue
flag :: Bool -> Flag a
flag = CheckedValue
getFlag :: Flag a -> Bool
getFlag = getCheckedValue

attribute :: String -> Attribute a
attribute = CheckedValue . Just
getAttribute :: Attribute a -> Maybe String
getAttribute = getCheckedValue

Constructing this CheckedValue datatype can be created generically, with the idea that its usages implement the assertion of whatever the specialized phantom will be filled with. I‚Äôll show what this means by reimplementing the truthy-parsing for flags, and the optional non-me authorship. 
checkMetadata
  :: a
  -> (String -> a)
  -> String
  -> Metadata
  -> CheckedValue a b
checkMetadata defaultValue f key meta =
  case lookupString key meta of
    Nothing -> CheckedValue defaultValue
    Just t -> CheckedValue (f t)

checkFlag :: String -> Metadata -> Flag a
checkFlag = checkMetadata False parsesTrue
 where
  parsesTrue t = map Char.toLower t `elem` ["yes", "true"]

checkAttribute :: (String -> Maybe String) -> String -> Metadata -> Attribute a
checkAttribute = checkMetadata Nothing

The specialized helpers ensure that usage sites can still limit what kind of scoped values are passed through, but even if it didn‚Äôt, the phantom type variables imply another barrier of entry before a function parameter is accepted.

One observation is that each type variable can encode one degree of functionality. I use the non-phantom variable, which I use to specialize to Flag and Attribute, to encode what kind of value is contained. While the phantom type variable encodes the key in the blogpost‚Äôs yaml that‚Äôs used to pull in the metadata value.

I‚Äôll finish this off with a showcase of value constructor functions and how easily they pull string data from the Metadata of a post into the typed world, applying any necessary conditions. 
isPublished :: Metadata -> Flag IsPublished
isPublished = checkFlag "published"

hasToC :: Metadata -> Flag IncludeTableOfContents
